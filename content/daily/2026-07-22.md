# 计算提速全记录：从热点优化到零全量落盘与多窗口复用

本文汇总 2026-07-11 至 2026-07-22 期间已经实施的全部“提速”工作。这里的“提速”不只指缩短 CPU 计算时间，也包括降低内存、减少落盘和重复查询、提高多日吞吐，以及让长任务能够断点恢复的工程改造。

日期以 Git 提交日期（Asia/Shanghai）为准。纯 telemetry、benchmark 和 guard 不会被当作算法提速，只作为定位瓶颈、验证结果和避免无效重跑的支撑措施。已经回退或被后续方案替代的实现会明确标注。

## 1. 总体结果

几轮优化形成了一条连续演进路线：

| 阶段 | 代表性结果 |
|---|---|
| 早期单股票日热点优化 | 9 个重股票日 factor build 从 `331.5s` 降至 `127.6s`，约 **2.6 倍提速**，180 个数值逐值一致 |
| shared-artifact 结构优化 | 全深市单日 train 从 `43m08s` 降至约 `25m59s`，主要消除了超线性扫描、重复事件处理和无条件工件构建 |
| 零全量落盘最终模式 | 单日从原始正式基线 `25m59s` 降至 `13m50s`，观察性缩短约 **46.8%**；相对同版本 full 模式缩短约 **28.1%** |
| 输出减重 | 最终同版本比较中，输出从 `4.57 GB / 6,724 files` 降至 `0.628 GB / 200 files`；候选边文件从 `6,588` 个降至 `0` |
| Doris 提取 | 全日顺序读取 `397.361s / 5.30 GB`，优化为 SQL 截断加三表有界并行后的 `117.723s / 2.35 GB` |
| 结果一致性 | 最终全日比较覆盖 `660,181,415` 条接受候选边；2,196 行面板的 1,221 个业务列逐值一致 |

上面的正式单日比较属于同口径观察，不是自动发布结论。当前安全默认值仍为 `full/store/legacy`；`sufficient/stream/fused` 已实现并通过完整日等价验证，但尚未自动切换为默认生产模式。

## 3. 2026-07-11：第一轮热点微优化

这一轮先用 cProfile 找出三个主要热点，再保持计算语义不变地替换 Python 慢路径。

### 3.1 生命周期构造：减少逐行和标量索引

原实现对订单使用 `iterrows()`，并在每只订单上反复调用 pandas `.loc` 查询撤单信息。改造后：

- 用 `itertuples()` 替代 `iterrows()`；
- 预先建立撤单数量、首次撤单时间、序号和位置字典；
- 循环内直接读取 Python 标量，避免反复触发 pandas 索引器和 Series 构造。

### 3.2 SRuns 链接：时间戳一次转换、整数比较

原实现会在候选向前扫描中重复构造 `pd.Timestamp`。改造后：

- 一次性把时间戳转为 int64 纳秒；
- 在扫描过程中直接比较整数；
- 把固定的 `link_window` 提出循环；
- 复用已经建立的 side、price 和 channel 字典，不再逐撤单读取 lifecycle 行。

### 3.3 有效价格：只给稀有情况保留逐行 fallback

限价单是常见路径，直接使用向量化的 `submit_price_tick / price_scale`；只有少量市价单或缺失价格行才调用逐行首笔成交价 fallback。

### 3.4 效果

- 9 个重股票日的 factor build：`331.5s → 127.6s`，约 **2.6 倍提速**；
- 12 个 attention 因子加 8 个辅助字段，共 `180/180` 个数值完全相同，最大绝对差为 `0`；
- 对应提交：`7ed013e`（2026-07-11 21:43）。

## 4. 2026-07-11～12：缓存、分片、断点续跑和按日并行

这一轮解决的是“每次都从头做”和“多天只能串行做”的问题。

### 4.1 2026-07-11：复用基础数据

- 把 1～3 日收益标签需要的复权开盘价、收盘价和停牌状态做成批量缓存，避免每个股票日重复联网或重复标准化；
- 复用按日期构造的 PIT 股票池和 L2 可得性；
- 面板按日写 Parquet shard；
- manifest 记录股票池、配置和输入指纹；
- 重跑时跳过指纹匹配的完整日期；
- 使用临时文件加原子替换，避免中断后把半成品当作成功结果。

对应提交包括 `7355819`、`b668223`，实施日期均为 2026-07-11。

### 4.2 2026-07-12：日期级多进程

- 新增 `ProcessPoolExecutor` 日级 runner；
- 每个交易日作为独立任务；
- 某一天失败只记录该日失败，不取消其他日期；
- 一 worker 与多 worker 生成的 shard 做字节级一致性验证；
- 结合 resume，已完成日期无需重新计算。

这一改造主要提高多日吞吐，并不缩短单个日期内部的股票计算。2025-07-01 全市场单 worker 基线为 `8054.765s`，即约 `2.237h`；四 worker 内存门测试未通过，随后根据机器内存约束采用更保守的并发数。对应提交为 `752dd24`（2026-07-12 16:28）。

## 5. 2026-07-17：把大任务改造成内存有界计算

服务器长任务出现内存风险后，优化重点从“多开 worker”转为“每个阶段只保留必要状态”。

### 5.1 calibration 候选流式处理

calibration 不再一次性把所有候选收集到一个大对象中，而是边读边校验、边累计统计量，限制候选常驻内存。

### 5.2 stage-2 两遍扫描

stage-2 与 idiosyncratic decomposition 改为两遍读取：第一遍建立必要的全局统计，第二遍生成结果。这样避免同时保留全部输入、分组中间量和输出。

### 5.3 历史 baseline 分类型构建

历史 baseline 不再同时构建所有类型，而是一次只构建一种并及时释放，降低峰值内存。

### 5.4 资源保护

增加 fail-closed memory guard、可配置 compute worker，以及对 cgroup OOM、`memory.max` 和无进度状态的记录。guard 本身不缩短正常计算，但能够在资源异常时尽早停止，避免数小时后 OOM、整段重跑。

主要提交：`e8596e1`、`3d47171`、`419f662`，实施日期为 2026-07-17。

## 6. 2026-07-18～19：从动态降级转向固定资源配置

### 6.1 2026-07-18：动态 worker 与分块重试

曾实现根据资源失败自动降低并发、按 chunk 重试和保存状态的 worker policy。这能让任务在未知资源条件下自动寻找可运行配置，对长时间批处理的有效吞吐有帮助。

主要提交为 `5efe93f`（2026-07-18 23:05）。

### 6.2 2026-07-19：固定并发和 watchdog 取代 worker ladder

动态升降会让不同运行之间难以比较，也可能反复试错。随后删除 worker ladder，改为：

- 生产命令使用显式、固定的 extract/compute/parser 参数；
- 用 CPU 配额检查阻止超额线程；
- 用 no-progress watchdog 和 cgroup 事件分类负责故障停止；
- 为单日任务提供独立 output root、无 resume 的 benchmark harness；
- benchmark 同时记录墙钟、CPU、内存、query、compute、输出体积和正确性 hash。

因此，2026-07-18 的动态 worker policy 属于历史尝试；当前保留的是固定资源、可观测、可回滚的执行方式。主要提交为 `0938dd0`、`405b563`、`1aa40a4`，实施日期为 2026-07-19。

## 7. 2026-07-20：候选边 NumPy/Arrow 分块重构

候选边原先大量依赖 pandas 对象、宽表和一次性中间结果。新版本把候选生成与评分改成有界、列式的数据流。

### 7.1 版本化的 `PairCandidateStore v2`

- 只保存必要的基础列；
- 派生列在消费时由 Arrow/NumPy 重建；
- schema、列顺序和 fingerprint 版本化；
- 兼容旧 v1 reader，允许已有工件继续读取。

### 7.2 搜索窗口与候选生成列式化

- 时间和事件窗口边界使用 `searchsorted` 定位；
- 使用 NumPy 数组生成 cancel/replacement 组合；
- 在物化前应用 side、segment、时间和因果边界；
- 避免 Python 双层循环和无效组合进入后续 overlap。

### 7.3 有界 RecordBatch 流

- 候选按固定上限切成 Arrow RecordBatch；
- batch 达到上限立即发送给消费者；
- 不要求把单只重股票的全部候选同时放入 DataFrame；
- 最终全日观测的最大 batch 为 `244,315` 行，始终满足 bound。

### 7.4 向量化评分和在线 reducer

- pair component score 改用 NumPy 列计算；
- local-best scorer 边读 batch 边更新最优项；
- grouped scorer 只保留有界分组状态；
- exact calibration 从候选 store 外置，避免为校准重复走完整构造链。

实施提交从 `8a558f6` 至 `079c3d9`，均完成于 2026-07-20。

## 8. 2026-07-20：正式调度和股票日级评分复用

候选边列式化之后，又对进程调度、历史查询和 scorer 之间的重复计算进行统一优化。

### 8.1 固定 16 worker 与 32 深度任务队列

- 使用长生命周期的 16 个 compute worker，不为每只股票重新创建进程；
- 保持最多 32 个在途任务，为 16 个 worker 提供两层缓冲；
- worker 内 BLAS/OpenMP/NumExpr 线程固定为 1，避免 `16 processes × N threads` 超额并行。

### 8.2 重股票优先与双日预取

- 根据原始行数或 Arrow bytes 估计股票任务重量；
- 使用 LPT 思路让重股票先进入队列，轻股票填补尾部，减少最后只剩少数慢股票的长尾；
- 多日时最多预取两个日期，让下一日抽取与本日计算重叠，同时禁止第三个日期进入内存。

### 8.3 历史 baseline 索引与 Arrow fast path

- 每个长期 worker 对同一日期、同一 baseline 文件只读取一次；
- baseline 读入后按股票建立内存索引；
- v2 候选直接投影为 Arrow 列，不再默认先还原成 pandas 宽表。

### 8.4 股票日级 `ScoringWorkspace`

在单个股票日内部共享：

- pair score；
- local-best/matching；
- standard/evidence chain；
- aggressive cluster；
- response/scorer 输出；
- aggregation 结果。

相同输入签名只计算一次，缓存只存在于当前股票日，不跨股票、不跨日期，也不缓存原始 L2。

### 8.5 聚合 NumPy 化和 trade-only label 查询

- count/qty/amount aggregation 共享只读 NumPy context；
- 相同 score 与 horizon 的聚合结果直接复用；
- 只需要成交价的标签链使用 trade-only QueryPlan，不再无条件读取 order 和 cancel。

主要实施提交为 `0f96bd7`（2026-07-20 21:37）。

## 9. 2026-07-21：shared-artifact 端到端结构优化

单日基准显示 shared-artifact 占 factor worker time 约 `91.3%`，所以这一轮不再增加 worker，而是降低每个股票日的实际计算量。

### 9.1 候选 overlap 向量化并提前剪枝

- 为 child events 建立复合查找结构；
- 不再按每个唯一 replacement 对整批 child events 重复扫描；
- 在候选物化前应用可判定的合法性边界；
- 保持接受候选集合、canonical order 和 fingerprint 不变。

### 9.2 不再回读刚写出的候选 Parquet

构建候选时同步收集 replacement IDs，直接交给 placebo 等下游消费者，不再“先写 Parquet，再为了 ID 读回来”。

### 9.3 canonical events 只标准化一次

order/trade/cancel 先形成一次稳定排序的 canonical event block，再由窗口、prestate 和 lifecycle 共享，减少重复类型转换、排序、concat 和 copy。

### 9.4 盘口深度增量维护

prestate 不再对每个 evidence event 重新全量排序和聚合订单簿，而是在事件回放过程中增量更新价格档位、FIFO 和 top depth。

### 9.5 impacts/placebo 列式化

market impacts 和 placebo sufficient 的逐行 pandas 处理改为 NumPy/Arrow 列式操作，减少对象创建和 groupby/apply 开销。

### 9.6 工件依赖图

每个 phase 和 active specification 显式声明需要的 artifact。训练阶段不再构建无人消费的 market impacts；已知会跳过的 specification 在进入重计算前就停止；每个 artifact 在单股票日最多构建一次。

主要提交为 `c3e553e`、`bea8a24`、`ee463a7`、`6046aa4`，完成于 2026-07-21 凌晨。单日 train 观察从 `43m08s` 降至约 `25m59s`。

## 10. 2026-07-21～22：候选边零全量落盘与有界流式流水线

这是当前幅度最大的一轮改造。目标从“更快生成候选文件”升级为“正式 sufficient 模式根本不生成全量候选文件”。

### 10.1 不可变 generation 与小 manifest 提交

大组件直接写入唯一 generation 路径，最终只发布小 manifest。这样避免大目录 rename 在对象存储上退化为复制再删除，也减少重复序列化和校验读回。

### 10.2 显式执行模式

增加并集中定义：

- train artifact：`full / shadow / sufficient`；
- pair scoring/storage：`store / shadow / stream`；
- event replay：`legacy / shadow / fused`。

`shadow` 用于同时运行新旧路径并比较结果；`sufficient/stream` 才是真正的零候选边全量落盘路径。

### 10.3 候选边多消费者在线处理

同一批 Arrow RecordBatch 在内存中依次交给：

- 候选流 summary；
- calibration sufficient-stat reducer；
- 在线 scorer/reducer；
- 仅在 full/shadow 模式存在的 Parquet sink。

在 `sufficient/stream` 下没有 Parquet sink，也不创建 pair generation、pair root 或 `.pair-candidate-staging`。

### 10.4 精确 calibration 在线充分统计

calibration 所需的 local-best、hash 和统计量在 batch 尚在内存时累计，不再依赖完整候选边文件。完整日的 2,196 份 calibration manifest 与 store 模式逐字段一致。

### 10.5 在线打分与批次特征复用

- 相同评分签名只算一次；
- 多个 scorer 共享同一份类型化特征视图；
- response market impact 建立索引，避免逐候选重复查找；
- 删除无界 score-batch cache；
- 活跃 scorer 审计路径中 Arrow→Pandas 次数为 `0`。

### 10.6 按日提交并立即释放

调度器增加 day-complete 回调。一天结束后立刻：

1. 完成 panel shard；
2. 归并 calibration/placebo 等派生片段；
3. 发布日级 manifest；
4. 删除已拥有的 scratch fragment；
5. 释放该日 payload，再进入下一天。

这样多日运行不再让父进程长期积累所有日期结果。

### 10.7 64 个确定性日级 bucket

逐股票的小工件按股票代码哈希归并到默认 64 个日级 bucket。文件数从跟随股票数增长改为跟随固定 bucket 数增长，同时保留按股票过滤读取和 fingerprint 校验能力。

### 10.8 lifecycle 与 prestate 融合回放

`fused` 模式在一次事件循环中共享状态，同时生成 lifecycle 与 prestate，避免两条链分别重放同一批事件。`shadow` 模式逐值比较 legacy 和 fused 输出后再允许使用 fused。

### 10.9 实施中的回退记录

2026-07-21 11:20～16:44 曾实施第一版工件控制面与零落盘方案，但在完整基线中发现控制面和性能问题，于 `da8adc5`（19:15）整体回退。最终版本从 `7e25756`（21:45）重新开始，经过 `b289788`、`8cc57d9`、`97f238b`、`26e8fe2`、`3ef45f5`、`452f091` 等提交重建并保留至当前 HEAD。本文的最终结果不把已回退版本计入有效成果。

## 11. 2026-07-22：Doris 查询裁剪与有界三表并行

原实现读取全天 order/trade/cancel，再在应用端裁剪到因子窗口。优化后把数据边界下推到 Doris：

- SQL 直接使用排他上界，例如 `data_time < '10:30:00'`；
- order、trade、cancel 最多三路并行；
- `table_parallelism × parser_threads` 不能超过 CPU 预算；
- telemetry 记录每表行数、字节数、首行时间、耗时和峰值内存；
- 应用端仍保留边界断言，防止错误 SQL 泄漏未来数据。

同一日期、同一 2,196 股票 universe 的观察结果：

| 读取方式 | 墙钟 | 客户端峰值内存 |
|---|---:|---:|
| 全天三表顺序读，再在应用端裁剪 | `397.361s` | `5.304 GB` |
| SQL `<10:30`，三表顺序读 | `152.965s` | `2.375 GB` |
| SQL `<10:30`，最多三表并行 | `117.723s` | `2.352 GB` |

三种方式裁剪后的 canonical hashes 完全一致。对应提交为 `0c3ca6e`（2026-07-22 02:43）。

## 12. 2026-07-22：多个因子窗口和标签共享一次 L2 提取

新增 09:30～09:33 因子窗口和日内 VWAP 标签后，如果每条链独立运行，会重复查询相同日期的 L2。最终实现改为：

- 根据本次启用的所有因子窗口和标签目标共同推导查询截止时间；
- 每日只创建一个 QueryPlan；
- 每张 L2 表只查询、解析一次；
- 同一份 `CodeDayInput` 依次交给 open1h、open3m 和标签 consumer；
- formal9 隔日标签与 intraday 标签在同一轮完成，不额外启动 standalone label scheduler；
- 不同窗口只共享 raw extraction，不共享会受窗口边界影响的 lifecycle、prestate、pair、placebo、theta 或 stage-2 工件；
- consumer 顺序运行，一个窗口完成后关闭 store、flush 并释放大对象，再进入下一窗口，避免峰值内存简单相加。

按照此前单日数据，这能避免每增加一条独立链约 `104s/日` 的重复 Doris 提取，同时降低数据库压力。该能力由 `ffb7cb3` 和 `50bfda6` 于 2026-07-22 实现，`c62db39` 完成本地多窗口正确性证明。服务器 A/B 性能栏目前仍待回填，因此不能把约 104 秒写成已经完成的服务器净提速。

## 14. 支撑提速但不单独计为“提速”的措施

以下工作没有直接减少业务计算量，但保证每轮优化可定位、可比较、可回滚：

- 将 `factor_seconds` 拆成 extraction、queue wait、worker execution、event conversion、prestate、lifecycle、pair enumeration、overlap、Arrow encode、Parquet write、placebo、scorer、aggregation、commit 等阶段；
- 记录 worker busy 数、任务队列深度、materialized day 数和 cache hit/miss；
- 记录每个 sink 的 consume 次数、行数、finish 时间和 retained bytes；
- 建立同日期、同 universe、无 resume、独立 output root 的 benchmark 纪律；
- 每次性能改造都用 schema、dtype、null、数值、canonical hash、candidate fingerprint、replacement IDs 和 manifest 做正确性闸门；
- 保留 full、shadow、sufficient 与 legacy、shadow、fused 的显式回滚开关。

这些 telemetry 还避免了一次无效优化：最终数据表明候选窗口 enumeration、overlap 和 Arrow encode 已不再是主要热点，因此没有为了“看起来更快”而缩短 300 秒或 100,000 event 的研究窗口。
